1. PROJECT OVERVIEW

1.1 The Business
Trevor Heaps Arboricultural Consultancy Ltd. (THAC) is a professional tree consultancy. The business surveys trees and produces formal written reports for clients who need professional arboricultural assessments. These reports are primarily used to support planning applications, discharge planning conditions, assess tree safety, manage insurance claims, or advise on development near trees.

THAC does not carry out physical tree works. Trevor is the consultant — he assesses, advises, and reports. The tree surgeon (the contractor who does the physical cutting, pruning, or felling) is a completely separate business engaged directly by the client and has no presence in this system.

1.2 The Problem
THAC currently operates from a 1,020-row Excel spreadsheet, Google Drive for document storage, and QuickBooks for invoicing. This works for 3–4 internal surveyors. It will not scale to a growing network of freelance surveyors, which is why this CRM is being built.

1.3 The Solution
A unified three-interface system — Client Portal, Admin CRM, Surveyor Mobile App — all powered by a single Supabase database. Data flows from a client's initial online enquiry through to a surveyor's mobile app, eliminating manual data entry and replacing the spreadsheet entirely.

3. THE THREE INTERFACES

3.1 Client Portal (Web — iframe embedded into trevorheaps.co.uk)
Public-facing. No login required. The client fills in a short enquiry form which branches based on whether the work is new or existing. The system auto-generates a quote and emails it. If accepted, a second detailed form captures full job information. Forms can also be completed internally by admin on behalf of a client (from a phone call, email, or web enquiry).

3.2 Admin Control Centre (Desktop Web App)
Used by Trevor and his admin team. The mission control interface for the entire operation. Contains two primary views of the same underlying data:
• The Urgency Dashboard — what jobs are due when, colour-coded by time remaining. Drives workload triage.
• The Live Map — what state each job is in, displayed as coloured dots at site coordinates. Drives surveyor dispatch.

3.3 Surveyor Field App (Mobile — iOS and Android)
Used by surveyors in the field. They only see the map interface — available jobs as red dots within their chosen mile radius from home. They tap a dot to preview a job, claim it, navigate to the exact site location, carry out the survey, and upload field data. They never see the admin triage, urgency tiers, or other surveyors' jobs.

9.2 Surveyor Pay Calculation
Formula: (1 hour travel + estimated survey time on site) × surveyor's personal hourly rate

• 1 hour travel is always paid regardless of actual distance to the site. Fixed and non-negotiable.
• Survey time on site is determined from a lookup table of hours per survey type (Trevor to provide). Admin can override this estimate at the approval gate.
• Each surveyor has their own hourly rate. The same job will display a different Ł value to different surveyors on the marketplace map.
• Urgent next-day jobs: surveyor earns standard calculated rate +20%. This percentage is stored as a configurable system value — not hardcoded — and can be adjusted without a code change.

9.3 Marketplace Visibility Rules
• A job is only visible to a surveyor on the marketplace map if the job's site coordinates fall within that surveyor's chosen mile radius from their home postcode.
• Radius filtering is handled server-side in Supabase using the Haversine formula (PostgreSQL earthdistance extension). Runs on every job-live event. No API cost, no external call, scales to any number of surveyors.
• When a surveyor taps a red dot, a single Google Distance Matrix API call fires and returns actual drive time and road distance from their current location to the job site. This is displayed on the job preview card to help them decide whether to claim.
• The radius is a soft filter — it stops irrelevant jobs appearing, not a hard commitment to attend. The surveyor hand picks what they want from what appears on their map.
• Pre-acceptance (red dot preview): surveyor sees survey type, Ł value to them, estimated tree numbers, required deadline. No client data.
• Post-acceptance (orange dot): full client details, site address, parking/access notes, and uploaded plans unlocked.
• Hand-back: surveyor can return a claimed job (due to sickness or inability to attend). Job returns to red dot and becomes available to other surveyors.

9.4 Radius Coverage — Technical Architecture (Option C)
The surveyor coverage radius system uses a hybrid approach — cheap server-side filtering combined with real drive-time data at the point of decision.

StageDetailOnboarding — postcode geocodingSurveyor enters home postcode. System calls Google Geocoding API once and converts postcode to lat/lng. Coordinates stored on surveyor profile in Supabase. Never needs repeating unless surveyor moves.Onboarding — radius settingSurveyor sets a single integer value in miles (e.g. 25). Stored on surveyor profile. Can be updated at any time.Job goes live — Haversine filterWhen Trevor approves a job, Supabase runs a Haversine distance query across all active surveyors. Any surveyor where the straight-line distance between their home lat/lng and the job's site lat/lng is within their radius sees the red dot appear on their map. 10. REFERENCE NUMBERS & AMENDMENTS

10.1 Reference Number Format
All jobs are assigned a sequential reference number in the format TH XXXX (e.g. TH 5291, TH 5292). The system generates this automatically on job creation.

10.2 Amendments
Amendments to existing reports are child records linked to the parent job. The reference appends a revision letter:
• First amendment: TH XXXX Rev B
• Second amendment: TH XXXX Rev C
• And so on.

Amendment pricing is time-and-materials at Ł100 + VAT per hour (minimum Ł25 + VAT). Admin records time taken. No automated quote is generated for amendments — they are instructed directly and billed after.

10.3 Two-Stage BS5837 Reports
Some BS5837 instructions are completed in two stages due to delays between the survey and the need for the final report (e.g. plans not yet finalised by the architect). These use staged 50/50 billing mode. The two jobs (Stage 1 survey, Stage 2 report) are linked records. Stage 2 sits in a 'waiting for plans' status until admin manually flips it to live, at which point it gets its own SLA timer and red dot on the map.

13. DATA IMPORT

Trevor's existing 1,020-row Excel spreadsheet will be imported into Supabase on go-live. Confirmed scope: Option B — live jobs plus completed/invoiced jobs. This preserves historical data for accounts purposes and enables returning-client recognition.

• Live jobs (status A, B, C, D, E, G in current spreadsheet) — imported as active records
• Sorted/invoiced jobs (status H) — imported as archived records, searchable but not on active dashboard
• Cancelled rows — not imported

A data migration script will be written in Sprint 1 to map the existing Excel columns to the new Supabase schema.
